# Optimised build decision

## Use this as the main build

Use the code in `../app/` as the main Repurly codebase.

## Why this build

It has the stronger:
- data model
- delivery and notification layer
- operator surfaces
- commercial upside
- future extensibility

## Strategic rule

Keep the architecture broad but the visible product narrow.

That means:
- use the richer backend
- market the product as a premium LinkedIn-first workflow tool
- avoid broad channel claims until the workflow is excellent

## Go / no-go lens

Do not call this truly launch-ready unless:
- the composer feels strong
- approvals are real and smooth
- scheduling works end to end
- activity / job detail are understandable
- reconnect guidance is useful
- demo / placeholder paths are not visible to customers

## Target customer

- agencies
- B2B marketing teams
- multi-brand or multi-account teams
- customers who value approvals, reliability, and visibility more than endless channel breadth
